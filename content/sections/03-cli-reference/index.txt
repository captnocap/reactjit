=== OVERVIEW ===
ReactJIT uses a CLI-first workflow. The `reactjit` command (aliased as `rjit`) wraps
esbuild with the correct flags, enforces lint gates before builds, handles runtime file
placement, and produces correct distribution packages for every platform.

Never run raw esbuild commands directly. The CLI encodes target-specific esbuild flags
(format, global name, externals, JSX mode), resolves entry point priorities, runs the
static linter before every build, and performs post-build bundle checks. Running esbuild
manually will use wrong flags, skip linting, and produce broken builds.

The CLI commands:

  rjit init <name>       Scaffold a new project with interactive package selection
  rjit dev               Watch mode with HMR -- rebuilds on file changes, launches Love2D
  rjit build [target]    Dev build (default) or production build for a specific platform
  rjit lint              Static layout linter -- catches mistakes before they reach the renderer
  rjit test <spec.ts>    Built-in test runner -- runs specs inside the Love2D process
  rjit screenshot        Headless capture -- verify layouts visually without manual inspection
  rjit update            Sync runtime files (lua/, lib/, reactjit/) from CLI into the project

Every build command (dev, build, screenshot) runs the linter first. Lint errors block
compilation. This is intentional -- layout mistakes caught statically are cheaper than
debugging visual artifacts at runtime.

=== EXAMPLES ===
Example 1: Typical development workflow
---
# Create a new project
rjit init my-app
cd my-app

# Start watch mode (rebuilds on save, launches Love2D)
rjit dev

# In another terminal: run the linter manually
rjit lint

# Take a screenshot for visual verification
rjit screenshot --output /tmp/preview.png

# Run tests
rjit test tests/my-app.test.ts

# Production build for Linux
rjit build linux
---
Platforms: love2d

=== CRITICAL RULES ===
- Never run raw esbuild commands -- always use rjit build, rjit dev, or rjit build <target>
- Lint errors block builds -- fix them before the bundle runs
- After editing framework source files (lua/, packages/core/, packages/renderer/), run make cli-setup then rjit update in consumer projects
- The storybook reads from source via symlinks -- never run rjit update from the storybook directory
