=== OVERVIEW ===
Comprehensive reference for the storage system's API surface: StorageProvider, useCRUD, createCRUD, useStorage, schema validation with z, query syntax, and migration configuration. All storage operations are async and return Promises. Schema validation is automatic on all reads and writes.

StorageProvider
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

<StorageProvider adapter={adapter}>
  {children}
</StorageProvider>

Props:
  adapter: StorageAdapter — Any adapter instance
  children: ReactNode

Purpose:
  Provides a default storage adapter to all child components.
  Components can access it via useStorage() or useCRUD() without
  passing { adapter } in options.

Example:
  const adapter = new MemoryAdapter();
  <StorageProvider adapter={adapter}>
    <App />
  </StorageProvider>

useStorage()
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

const adapter = useStorage();

Returns: StorageAdapter

Purpose:
  Returns the storage adapter from StorageProvider context.
  Throws if used outside StorageProvider.

Methods on StorageAdapter:
  get(collection, id) → Promise<any | null>
  set(collection, id, data) → Promise<void>
  delete(collection, id) → Promise<boolean>
  list(collection, query?) → Promise<any[]>
  format?: 'json' | 'markdown' | 'text'

Example:
  const adapter = useStorage();
  const raw = await adapter.get('users', 'u1');
  await adapter.set('users', 'u2', { name: 'Bob' });

useCRUD(collection, schema, options?)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

const crud = useCRUD<T>(collection, schema, options?);

Parameters:
  collection: string — Collection name (e.g. 'users', 'notes')
  schema: Schema<T> — Validation schema (z.object(...))
  options?: CRUDOptions
    adapter?: StorageAdapter — Override default from context
    format?: 'json' | 'markdown' | 'text'
    migrations?: Record<number, MigrationFn>
    autoMigrate?: boolean

Returns: CRUDHandle<T> with methods:
  create(data: T) → Promise<string>
  get(id: string) → Promise<T | null>
  update(id: string, partial: Partial<T>) → Promise<void>
  delete(id: string) → Promise<void>
  list(query?: Query) → Promise<T[]>
  useQuery(id: string) → { data, loading, error, refetch }
  useListQuery(query?: Query) → { data, loading, error, refetch }

CRUD Method Details:

create(data: T) → Promise<string>
  - Validates data against schema
  - Generates ID if not provided (format: timestamp-random-seq)
  - Saves to storage
  - Returns the ID

get(id: string) → Promise<T | null>
  - Fetches record by ID
  - Validates against schema
  - Returns null if not found

update(id: string, partial: Partial<T>) → Promise<void>
  - Fetches existing record
  - Merges with partial data
  - Validates merged result
  - Saves back to storage
  - Throws if record not found

delete(id: string) → Promise<void>
  - Removes record from storage
  - No-op if record doesn't exist

list(query?: Query) → Promise<T[]>
  - Fetches all records in collection
  - Applies query filters (where, orderBy, limit, offset)
  - Validates each record against schema
  - Returns array (empty if no matches)

useQuery(id: string)
  - Reactive hook for single record
  - Returns { data, loading, error, refetch }
  - data: T | null
  - loading: boolean
  - error: Error | null
  - refetch: () => void
  - Automatically re-fetches when id changes
  - Does NOT auto-refetch on mutations — call refetch()

useListQuery(query?: Query)
  - Reactive hook for collection query
  - Returns { data, loading, error, refetch }
  - data: T[]
  - loading: boolean
  - error: Error | null
  - refetch: () => void
  - Does NOT auto-refetch on mutations — call refetch()

Example:
  const notes = useCRUD('notes', NoteSchema);

  // Create
  const id = await notes.create({ title: 'Hello', body: 'World' });

  // Read
  const note = await notes.get(id);

  // Update
  await notes.update(id, { title: 'Updated' });

  // Delete
  await notes.delete(id);

  // List
  const all = await notes.list();
  const filtered = await notes.list({ where: { archived: false } });

  // Reactive
  const { data, loading, refetch } = notes.useQuery(id);
  const { data: list } = notes.useListQuery();

createCRUD(collection, schema, adapter, options?)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

const crud = createCRUD<T>(collection, schema, adapter, options?);

Parameters:
  collection: string
  schema: Schema<T>
  adapter: StorageAdapter — Required (no context)
  options?: CreateCRUDOptions
    migrations?: Record<number, MigrationFn>
    autoMigrate?: boolean

Returns: CRUDMethods<T> (same as useCRUD but without reactive hooks)
  create(data: T) → Promise<string>
  get(id: string) → Promise<T | null>
  update(id: string, partial: Partial<T>) → Promise<void>
  delete(id: string) → Promise<void>
  list(query?: Query) → Promise<T[]>

Purpose:
  Non-React CRUD interface for scripts, CLI tools, and backends.
  Same validation and migration support as useCRUD.
  No hooks, no context — pass adapter explicitly.

Example:
  import { createCRUD, z, MemoryAdapter } from '@reactjit/storage';

  const adapter = new MemoryAdapter();
  const users = createCRUD('users', UserSchema, adapter);

  await users.create({ id: 'u1', name: 'Alice' });
  const all = await users.list();

Schema Validation (z)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

import { z } from '@reactjit/storage';

Primitive Types:
  z.string()
  z.number()
  z.boolean()
  z.date()

Complex Types:
  z.object({ field1: z.string(), field2: z.number() })
  z.array(z.string())

Modifiers:
  .optional() — allows undefined
  .nullable() — allows null
  .default(value) — uses value if undefined (implies optional)

Schema Methods:
  parse(value) → T — throws ValidationError if invalid
  safeParse(value) → SafeParseResult<T>
    Success: { success: true, data: T }
    Failure: { success: false, error: ValidationError }

ValidationError Properties:
  issues: ValidationIssue[]
  message: string (formatted from issues)

ValidationIssue:
  path: string[] — field path (e.g. ['user', 'email'])
  message: string — error description
  code: string — error code (e.g. 'invalid_type')

Example:
  const UserSchema = z.object({
    id: z.string(),
    name: z.string(),
    age: z.number().optional(),
    bio: z.string().nullable(),
    role: z.string().default('user'),
    tags: z.array(z.string()),
  });

  // Parse (throws on error)
  const user = UserSchema.parse(rawData);

  // Safe parse (returns result object)
  const result = UserSchema.safeParse(rawData);
  if (result.success) {
    console.log(result.data);
  } else {
    console.error(result.error.issues);
  }

Type Inference:
  type User = z.infer<typeof UserSchema>;
  // Infers: { id: string; name: string; age?: number; bio: string | null; role: string; tags: string[]; }

Query Syntax
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

interface Query {
  where?: WhereClause;
  orderBy?: string;
  order?: 'asc' | 'desc';
  limit?: number;
  offset?: number;
}

type WhereClause = Record<string, any | Partial<Record<ComparisonOp, any>>>;
type ComparisonOp = '$eq' | '$ne' | '$gt' | '$gte' | '$lt' | '$lte' | '$in' | '$contains';

Query Operators:

Equality:
  { field: value } — exact match (shorthand for $eq)
  { field: { $eq: value } } — exact match (explicit)
  { field: { $ne: value } } — not equal

Comparison:
  { field: { $gt: value } } — greater than
  { field: { $gte: value } } — greater than or equal
  { field: { $lt: value } } — less than
  { field: { $lte: value } } — less than or equal

Array/String:
  { field: { $in: [val1, val2] } } — value in array
  { field: { $contains: substring } } — string contains substring

Multiple Conditions:
  { field1: value1, field2: { $gt: 10 } } — AND logic
  All conditions must match

Sorting:
  orderBy: 'fieldName'
  order: 'asc' or 'desc' (default: 'asc')

Pagination:
  limit: number — max records to return
  offset: number — skip first N records

Examples:

// Exact match
await crud.list({ where: { status: 'active' } });

// Greater than
await crud.list({ where: { age: { $gt: 18 } } });

// Range
await crud.list({ where: { price: { $gte: 10, $lte: 100 } } });

// In array
await crud.list({ where: { category: { $in: ['electronics', 'books'] } } });

// Contains substring
await crud.list({ where: { title: { $contains: 'react' } } });

// Multiple conditions
await crud.list({
  where: { status: 'published', views: { $gt: 1000 } }
});

// Sort and paginate
await crud.list({
  where: { archived: false },
  orderBy: 'createdAt',
  order: 'desc',
  limit: 20,
  offset: 40,
});

Migrations
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

type MigrationFn = (data: any) => any;

interface MigrationConfig {
  migrations?: Record<number, MigrationFn>;
  autoMigrate?: boolean;
}

Pass to useCRUD or createCRUD options:
  migrations: Record<number, MigrationFn>
    Keys: version numbers (1, 2, 3, ...)
    Values: functions that transform data

  autoMigrate: boolean
    true — migrations run automatically on every read
    false — migrations must be triggered manually

Migration Execution:
  - Runs on get() and list() operations
  - Applies migrations in version order (1 → 2 → 3)
  - Stores __version field in data to track migration state
  - Each migration receives output of previous migration
  - Migrations are not saved back to storage unless record is updated

Best Practices:
  - Keep migration functions pure and idempotent
  - Add default values for new fields
  - Rename/transform fields as needed
  - Avoid async operations in migration functions
  - Test migrations with real data before deploying

Example:
  const UserSchema = z.object({
    id: z.string(),
    fullName: z.string(),
    email: z.string(),
    preferences: z.object({
      theme: z.string(),
      notifications: z.boolean(),
    }),
  });

  const users = useCRUD('users', UserSchema, {
    migrations: {
      // v1: Add email field
      1: (data) => ({
        ...data,
        email: data.email || `${data.username}@example.com`,
      }),

      // v2: Rename 'name' to 'fullName'
      2: (data) => {
        const { name, ...rest } = data;
        return { ...rest, fullName: name || 'Unknown' };
      },

      // v3: Add preferences object
      3: (data) => ({
        ...data,
        preferences: data.preferences || {
          theme: 'dark',
          notifications: true,
        },
      }),
    },
    autoMigrate: true,
  });

  // Old records are migrated on read
  const user = await users.get('old-user-id');
  // user.__version === 3

Manual Migration Trigger:
  // autoMigrate: false
  const adapter = new MemoryAdapter();
  const migrating = createMigratingAdapter(adapter, migrations, false);
  await migrating.migrate('users'); // Migrates all records in collection

Adapters

All adapters implement the StorageAdapter interface:
  get(collection, id) → Promise<any | null>

─────────────────────────────────────────────────────────
MemoryAdapter
─────────────────────────────────────────────────────────

new MemoryAdapter()

No options. Data stored in Map<string, Map<string, any>>.
Data lost when app closes.

Use for:
  - Testing
  - Prototyping
  - Ephemeral state
  - Targets without persistent storage

Example:
  const adapter = new MemoryAdapter();

─────────────────────────────────────────────────────────
Love2DFileAdapter
─────────────────────────────────────────────────────────

new Love2DFileAdapter({ rpc, format? })

Options:
  rpc: (method, args?) => Promise<T> — RPC caller (from useLoveRPC)
  format?: 'json' | 'markdown' | 'text' — Default: 'json'

Storage Location:
  save/<collection>/<id>.<ext>
  (in Love2D's save directory)

RPC Methods Called:
  storage:get — read file
  storage:set — write file
  storage:delete — delete file
  storage:list — list files in collection

Formats:
  json — JSON.parse/stringify (best for structured data)
  markdown — .md extension (best for text/notes)
  text — plain text (best for logs)

Example:
  import { useLoveRPC } from '@reactjit/core';
  const rpc = useLoveRPC();
  const adapter = new Love2DFileAdapter({ rpc, format: 'json' });

─────────────────────────────────────────────────────────
LocalStorageAdapter
─────────────────────────────────────────────────────────

new LocalStorageAdapter({ prefix? })

Options:
  prefix?: string — Key prefix (default: 'ilr:')

Storage Key Format:
  <prefix><collection>:<id>
  Example: 'ilr:notes:n1'

Limits:
  - 5-10MB total (browser-dependent)
  - Synchronous API (may block on large operations)
  - No structured queries (all done in-memory)

Best for:
  - Small datasets
  - Simple key-value storage
  - User preferences

Example:
  const adapter = new LocalStorageAdapter({ prefix: 'myapp:' });

─────────────────────────────────────────────────────────
IndexedDBAdapter
─────────────────────────────────────────────────────────

new IndexedDBAdapter({ dbName?, version? })

Options:
  dbName?: string — Database name (default: 'reactjit')
  version?: number — Schema version (default: 1)

Storage:
  - One object store per collection
  - Records keyed by 'id' field
  - Async transactional API

Limits:
  - Much larger than localStorage (50MB+)
  - Structured queries run in-memory (no indexes)

Best for:
  - Larger datasets
  - Complex data structures
  - Offline-first apps

Example:
  const adapter = new IndexedDBAdapter({ dbName: 'taskdb', version: 1 });

─────────────────────────────────────────────────────────
TerminalSQLiteAdapter
─────────────────────────────────────────────────────────

new TerminalSQLiteAdapter({ dbPath? })

Options:
  dbPath?: string — Database file path (default: './data.db')

Storage:
  - One table per collection
  - Schema: id TEXT PRIMARY KEY, data TEXT (JSON)
  - Uses node:sqlite (Node.js v22+)

Queries:
  - WHERE clauses compile to SQL
  - ORDER BY, LIMIT, OFFSET use SQL
  - Faster than in-memory filtering

Requirements:
  - Node.js v22 or later
  - Does not work in browsers

Best for:
  - Terminal apps
  - CLI tools
  - Large datasets with structured queries

Example:
  const adapter = new TerminalSQLiteAdapter({ dbPath: './data.db' });

Example 2: Manual Refetch on Mutation
---
function TodoList() {
  const todos = useCRUD('todos', TodoSchema);
  const { data, loading, refetch } = todos.useListQuery();

  const handleAdd = async () => {
    await todos.create({ text: 'New todo', completed: false });
    refetch(); // REQUIRED — reactive hooks don't auto-refetch
  };

  const handleToggle = async (id: string) => {
    const todo = await todos.get(id);
    if (todo) {
      await todos.update(id, { completed: !todo.completed });
      refetch(); // REQUIRED
    }
  };

  return (
    <Box>
      {data.map(todo => (
        <Pressable key={todo.id} onPress={() => handleToggle(todo.id)}>
          <Text fontSize={14}>{todo.text}</Text>
        </Pressable>
      ))}
      <Pressable onPress={handleAdd}>
        <Text fontSize={14}>+ Add Todo</Text>
      </Pressable>
    </Box>
  );
}

Example 5: Error Handling with safeParse
---
function SafeCreateForm() {
  const users = useCRUD('users', UserSchema);
  const [error, setError] = useState<string | null>(null);

  const handleSubmit = async (formData: any) => {
    const result = UserSchema.safeParse(formData);
    if (!result.success) {
      const messages = result.error.issues.map(i => i.message).join(', ');
      setError(messages);
      return;
    }

    try {
      await users.create(result.data);
      setError(null);
    } catch (e) {
      setError(e.message);
    }
  };

  return (
    <Box>
      {error && <Text fontSize={12} color="red">{error}</Text>}
      {/* form fields */}
    </Box>
  );
}

Example 6: Non-React Script with SQLite

#!/usr/bin/env node
import { createCRUD, z, TerminalSQLiteAdapter } from '@reactjit/storage';

const TaskSchema = z.object({
  id: z.string(),
  text: z.string(),
  completed: z.boolean(),
  createdAt: z.date(),
});

const adapter = new TerminalSQLiteAdapter({ dbPath: './tasks.db' });
const tasks = createCRUD('tasks', TaskSchema, adapter);

async function main() {
  // Add task
  await tasks.create({
    id: `t${Date.now()}`,
    text: process.argv[2] || 'New task',
    completed: false,
    createdAt: new Date(),
  });

  // List all
  const all = await tasks.list({ orderBy: 'createdAt', order: 'desc' });
  all.forEach(t => console.log(`[${t.completed ? 'x' : ' '}] ${t.text}`));
}

main();

Example 7: Multi-Collection App
---
const NoteSchema = z.object({ id: z.string(), title: z.string(), body: z.string() });
const TagSchema = z.object({ id: z.string(), name: z.string(), color: z.string() });
const AttachmentSchema = z.object({ id: z.string(), noteId: z.string(), url: z.string() });

function App() {
  const rpc = useLoveRPC();
  const adapter = new Love2DFileAdapter({ rpc });

  return (
    <StorageProvider adapter={adapter}>
      <NoteManager />
    </StorageProvider>
  );
}

function NoteManager() {
  const notes = useCRUD('notes', NoteSchema);
  const tags = useCRUD('tags', TagSchema);
  const attachments = useCRUD('attachments', AttachmentSchema);

  const { data: noteList } = notes.useListQuery();
  const { data: tagList } = tags.useListQuery();

  // Each collection is independent but uses the same adapter
  return (
    <Box>
      <Box>
        {noteList.map(note => (
          <Text key={note.id} fontSize={14}>{note.title}</Text>
        ))}
      </Box>
      <Box>
        {tagList.map(tag => (
          <Text key={tag.id} fontSize={12}>{tag.name}</Text>
        ))}
      </Box>
    </Box>
  );
}

=== PLATFORM NOTES ===
Love2D: Love2DFileAdapter requires RPC connection (useLoveRPC). Files stored in Love2D's save directory (OS-dependent). Supports JSON, Markdown, Text formats. Query filtering happens in-memory after fetching all records. Large collections may be slow.

Web: LocalStorageAdapter is synchronous and limited to 5-10MB total. IndexedDBAdapter is async, supports larger datasets, and uses transactions. Both adapters filter queries in-memory. For production apps with >1000 records, prefer IndexedDB.

Terminal: TerminalSQLiteAdapter requires Node.js v22+ for node:sqlite. Query filtering compiles to SQL WHERE clauses for efficient filtering. Supports millions of records with proper indexing (manual via raw SQL). Use for CLI tools, terminal UIs, and server-side scripts.

MemoryAdapter: Works everywhere but loses data on close. Perfect for testing and prototyping. No persistence, no file I/O, no network.

=== CRITICAL RULES ===
- Schema MUST include `id: z.string()` field — CRUD methods depend on it
- Reactive hooks (useQuery, useListQuery) do NOT auto-refetch — call refetch() after mutations
- Love2DFileAdapter requires useLoveRPC() — pass rpc function to constructor
- TerminalSQLiteAdapter only works in Node.js v22+ — not available in browsers
- IndexedDBAdapter operations are async — always await
- LocalStorageAdapter has 5-10MB total limit across all collections
- Query operators ($gt, $in, $contains) work in-memory for most adapters except TerminalSQLiteAdapter (uses SQL)
- Migration functions must be pure and idempotent — no async operations
- autoMigrate: true runs migrations on every read — keep migration logic fast
- Schema validation runs on ALL reads and writes — throws ValidationError on failure
- Do not mutate data returned from get() or list() — it may be cached
- Use createCRUD for scripts/backends without React — useCRUD is for React components only
- StorageProvider must wrap your app if you don't pass { adapter } to useCRUD
- Generated IDs are globally unique but NOT guaranteed to be sortable — use timestamps if you need chronological ordering
- delete() is idempotent — calling it on a non-existent ID does not throw
