=== OVERVIEW ===
Every frame in ReactJIT follows a fixed sequence. Love2D owns the run loop -- React never
drives the frame. Understanding this pipeline is essential for debugging layout issues,
performance problems, and event timing.

The frame cycle:

  love.update(dt)
    1. Lua calls JS: globalThis._pollAndDispatchEvents()
       - JS reads queued events from Lua via __hostGetEvents()
       - Events dispatch to React handlers via the handlerRegistry
       - JS animations tick (useAnimation, useSpring, etc.)
    2. Handlers may trigger React state updates
       - React re-renders affected components
       - The reconciler diffs old vs new props
       - Mutation commands (CREATE, UPDATE, REMOVE, APPEND) are emitted
    3. resetAfterCommit() calls flushToHost()
       - Commands are coalesced (multiple UPDATEs to the same node merge)
       - JSON-serialized and sent to Lua via __hostFlush()
    4. Lua applies commands to the instance tree (tree.lua)
       - Nodes are created, updated, reparented, or removed
       - Tree is marked dirty
    5. Lua runs layout (layout.lua)
       - Computes {x, y, w, h} for every node
       - Text measurement uses Love2D's font rasterizer
       - Only runs when the tree is dirty

  love.draw()
    6. Lua paints the tree (painter.lua)
       - Walks nodes in tree order, respecting zIndex for paint ordering
       - Issues Love2D graphics calls: rectangles, text, images, stencils
       - Applies opacity, clipping, transforms, shadows, gradients

This means React state changes are visible in the SAME FRAME they occur. There is no
double-buffering delay between React and rendering. The sequence is synchronous within
a single love.update/love.draw cycle.

Event handlers never fire asynchronously. Lua queues events, JS processes them when polled,
React re-renders, commands flush, tree updates, layout runs -- all within love.update().
The paint pass in love.draw() sees the fully updated tree.

=== API / SYNTAX ===
The pipeline stages correspond to these source files:

  React reconciler:
    packages/renderer/src/hostConfig.ts    -- Host config, instance creation, mutation commands
    packages/renderer/src/NativeBridge.ts  -- Bridge transport, event polling, RPC

  QuickJS FFI bridge:
    lua/bridge_quickjs.lua                 -- C API bindings, host functions, value conversion

  Instance tree:
    lua/tree.lua                           -- Retained node tree, command interpreter

  Layout engine:
    lua/layout.lua                         -- Flexbox computation, text measurement, unit resolution

  Painter:
    lua/painter.lua                        -- Love2D draw calls, text rendering, clipping, effects

  Integration:
    lua/init.lua                           -- Wires everything together, owns the frame loop

Mutation command types (emitted by the reconciler, consumed by tree.lua):

  CREATE        -- Create a new node with type and props
  CREATE_TEXT   -- Create a text leaf node
  APPEND        -- Append a child to a parent
  APPEND_TO_ROOT -- Append a child to the root container
  UPDATE        -- Update props on an existing node (partial diff)
  UPDATE_TEXT   -- Update text content on a text node
  REMOVE        -- Remove a child from a parent (and clean up descendants)
  REMOVE_FROM_ROOT -- Remove a child from the root container
  INSERT_BEFORE -- Insert a child before a sibling

=== EXAMPLES ===
Example 1: A component's journey through the pipeline
---
import { Box, Text } from '@reactjit/core';
import { useThemeColors } from '@reactjit/theme';

// This component declaration triggers the full pipeline:
function Greeting({ name }: { name: string }) {
  const c = useThemeColors();
  return (
    <Box style={{ padding: 16, backgroundColor: c.surface, borderRadius: 8 }}>
      <Text style={{ fontSize: 18, color: c.text }}>{`Hello, ${name}!`}</Text>
    </Box>
  );
}

// React reconciler emits:
//   CREATE { id: 1, type: 'View', props: { style: { padding: 16, ... } } }
//   CREATE { id: 2, type: 'Text', props: { style: { fontSize: 18, ... } } }
//   CREATE_TEXT { id: 3, text: 'Hello, World!' }
//   APPEND { parentId: 2, childId: 3 }
//   APPEND { parentId: 1, childId: 2 }
//   APPEND_TO_ROOT { childId: 1 }
//
// tree.lua applies these commands to build:
//   root -> View(1) -> Text(2) -> __TEXT__(3)
//
// layout.lua computes positions:
//   View(1):  { x: 0, y: 0, w: 1280, h: 46 }
//   Text(2):  { x: 16, y: 16, w: 1248, h: 14 }
//   __TEXT__(3): inherits from Text(2)
//
// painter.lua draws:
//   1. Rounded rectangle at (0,0) 1280x46 with surface color
//   2. Text "Hello, World!" at (16,16) with text color
---
Platforms: love2d

Example 2: How a state update flows through the pipeline
---
import { useState } from 'react';
import { Box, Text, Pressable } from '@reactjit/core';
import { useThemeColors } from '@reactjit/theme';

function Counter() {
  const c = useThemeColors();
  const [count, setCount] = useState(0);

  return (
    <Box style={{ padding: 16 }}>
      <Pressable onClick={() => setCount(count + 1)}>
        <Text style={{ fontSize: 16, color: c.text }}>{`Count: ${count}`}</Text>
      </Pressable>
    </Box>
  );
}

// When the user clicks:
// 1. Lua detects mouse press, hit-tests the Pressable node
// 2. Lua queues a 'click' event with targetId and bubblePath
// 3. Next frame: JS polls events, dispatches onClick to the handler
// 4. setCount(1) triggers React re-render
// 5. Reconciler diffs: only the text changed
// 6. Emits: UPDATE_TEXT { id: 3, text: 'Count: 1' }
// 7. tree.lua updates node 3's text
// 8. layout.lua re-measures text width
// 9. painter.lua draws the updated text
// Total: one frame from click to visible update
---
Platforms: love2d

=== CRITICAL RULES ===
- Love2D owns the run loop. React never calls requestAnimationFrame or setTimeout to drive rendering. Lua calls into JS each frame; JS responds synchronously.
- Event handlers fire during love.update(), not asynchronously. By the time love.draw() runs, all state changes from the current frame's events are already applied.
- Mutation commands are coalesced before crossing the bridge. Multiple UPDATEs to the same node within one commit merge into a single command. This reduces bridge traffic.
- Commands are JSON-serialized to avoid a QuickJS GC race condition where large string properties get silently dropped during FFI object traversal.
