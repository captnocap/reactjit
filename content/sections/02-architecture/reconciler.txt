=== OVERVIEW ===
ReactJIT uses react-reconciler (v0.29) with a custom host config. The reconciler diffs your
component tree and emits mutation commands that describe what changed. These commands are the
only thing that crosses the bridge to Lua. React never directly touches the instance tree,
the layout engine, or the painter.

The host config lives in packages/renderer/src/hostConfig.ts. It implements the mutation mode
(supportsMutation: true) -- no persistent mode, no hydration.

Key design decisions:

Handler separation. Event handlers (onClick, onKeyDown, etc.) never cross the bridge. They
stay in a JavaScript-side handlerRegistry keyed by node ID. When Lua detects a user
interaction, it sends an event with a targetId. The event dispatcher looks up handlers in the
registry and calls them directly. Only serializable props (style, text content, flags) are
sent to Lua via mutation commands.

Partial diffing. The reconciler does not send full props on every update. prepareUpdate()
computes a minimal diff between old and new props, including partial style diffs. Only changed
style keys are sent. Removed keys are tracked separately so Lua can nil them out.

Command coalescing. Multiple UPDATEs to the same node within one React commit are merged into
a single command before flushing. This reduces bridge traffic when a component updates several
props across multiple effects or callbacks in the same frame.

HTML remapping. For compatibility, standard HTML element types (div, span, h1, img, etc.) are
automatically remapped to ReactJIT host types (View, Text, Image). This happens transparently
in createInstance before the CREATE command is emitted. Lua never sees unknown types.

=== API / SYNTAX ===
Instance structure (JS side):

  interface Instance {
    id: number;           // Unique node ID, monotonically increasing
    type: string;         // Host type: 'View', 'Text', 'Image', etc.
    props: Record;        // Clean props (no handlers, no children)
    handlers: Record;     // Event handlers (onClick, onKeyDown, etc.)
    children: Instance[]; // Child instances
    renderCount: number;  // How many times this instance has been committed
  }

Mutation commands emitted by the reconciler:

  CREATE { id, type, props, hasHandlers, handlerMeta?, debugName?, debugSource? }
    Creates a new node. props are clean (handlers extracted). hasHandlers tells Lua whether
    this node should participate in hit testing. handlerMeta provides function source snippets
    for the visual inspector.

  CREATE_TEXT { id, text }
    Creates a text leaf node (the __TEXT__ type in Lua).

  APPEND { parentId, childId }
    Appends a child to a parent node.

  APPEND_TO_ROOT { childId }
    Appends a child to the root container.

  UPDATE { id, props, hasHandlers, handlerMeta?, removeKeys?, removeStyleKeys?, renderCount? }
    Updates an existing node. props contains only changed values. style is a partial diff
    (only changed keys). removeKeys lists top-level props that were removed. removeStyleKeys
    lists style properties that were removed.

  UPDATE_TEXT { id, text }
    Updates text content on a text leaf node.

  REMOVE { parentId, childId }
    Removes a child from its parent and cleans up all descendants.

  REMOVE_FROM_ROOT { childId }
    Removes a child from the root container.

  INSERT_BEFORE { parentId, childId, beforeId }
    Inserts a child before a sibling (for reordering).

Handler registry:

  The handlerRegistry is a Map<number, Record<string, Function>> that maps node IDs to their
  event handlers. When a node is created or updated, extractHandlers() separates on* function
  props from serializable props. Handlers are stored in the registry; clean props are sent
  to Lua. When a node is removed, its handlers (and all descendant handlers) are cleaned up.

=== EXAMPLES ===
Example 1: How createInstance works
---
import { Box, Text, Pressable } from '@reactjit/core';
import { useThemeColors } from '@reactjit/theme';

function Card({ title, onPress }: { title: string; onPress: () => void }) {
  const c = useThemeColors();
  return (
    <Pressable onClick={onPress}>
      <Box style={{ padding: 16, backgroundColor: c.surface, borderRadius: 8 }}>
        <Text style={{ fontSize: 16, color: c.text }}>{`${title}`}</Text>
      </Box>
    </Pressable>
  );
}

// The reconciler calls createInstance for each host element:
//
// 1. Pressable -> createInstance('View', { onClick: fn })
//    extractHandlers separates: clean={}, handlers={ onClick: fn }
//    Emits: CREATE { id:1, type:'View', props:{}, hasHandlers:true }
//    handlerRegistry.set(1, { onClick: fn })
//
// 2. Box -> createInstance('View', { style: { padding:16, ... } })
//    No handlers.
//    Emits: CREATE { id:2, type:'View', props:{ style:{...} }, hasHandlers:false }
//
// 3. Text -> createInstance('Text', { style: { fontSize:16, ... } })
//    Emits: CREATE { id:3, type:'Text', props:{ style:{...} }, hasHandlers:false }

Example 2: Partial prop diffing on update
---
import { useState } from 'react';
import { Box } from '@reactjit/core';
import { useThemeColors } from '@reactjit/theme';

function ToggleBox() {
  const c = useThemeColors();
  const [active, setActive] = useState(false);

  // When active changes from false to true:
  // prepareUpdate diffs the style objects:
  //   old: { padding: 16, backgroundColor: c.surface }
  //   new: { padding: 16, backgroundColor: c.primary }
  // Result: { diff: { style: { backgroundColor: c.primary } }, removeKeys: [], removeStyleKeys: [] }
  // Only the changed backgroundColor is sent to Lua -- padding is unchanged and skipped.
  return (
    <Box
      style={{
        padding: 16,
        backgroundColor: active ? c.primary : c.surface,
        borderRadius: 8,
      }}
    />
  );
}

=== CRITICAL RULES ===
- Event handlers NEVER cross the bridge. They stay in the JS handlerRegistry. If you add a new event type, you must add it to the Box destructure list in primitives.tsx AND subscribe to it in eventDispatcher.ts. Skipping this causes handlers to silently disappear.
- Props are partially diffed. Only changed values are sent on UPDATE. Do not assume Lua receives the full prop object on every update.
- The hasHandlers flag on CREATE/UPDATE tells Lua whether a node should participate in hit testing. If handlers are removed from a node, hasHandlers is set to false and Lua stops hit-testing it.
- Node IDs are monotonically increasing integers. They are never reused within a session.
