=== OVERVIEW ===
These are the remaining critical rules in ReactJIT development. The layout engine has been
refined to handle many cases that previously required strict rules: Text now has sensible
defaults, Unicode symbols render correctly, rows resolve width from parents, deep flex nesting
works, and large component trees handle correctly. Only 4 rules remain because they reflect
fundamental constraints of the layout system.

The linter runs automatically during `reactjit build` (lint gate). You should also run
`reactjit lint` after every component change to catch issues early. Rules can be bypassed
on a per-line basis with the `// rjit-ignore-next-line` comment, but this should be rare.

=== API / SYNTAX ===
Linter escape hatch:

  // rjit-ignore-next-line
  <Text style={{ color: '#cdd6f4' }}>Intentionally no fontSize</Text>

This suppresses all linter rules for the next line. Use sparingly.

=== EXAMPLES ===
Example 1: WRONG -- root uses flexGrow
---
// This will NOT fill the viewport. flexGrow on the root has no parent to grow into.
function App() {
  return (
    <Box style={{ flexGrow: 1, backgroundColor: '#1e1e2e' }}>
      <Text style={{ fontSize: 20, color: '#cdd6f4' }}>Invisible</Text>
    </Box>
  );
}
---
Platforms: All

Example 2: RIGHT -- root uses explicit dimensions
---
function App() {
  return (
    <Box style={{ width: '100%', height: '100%', backgroundColor: '#1e1e2e' }}>
      <Text style={{ fontSize: 20, color: '#cdd6f4' }}>Visible</Text>
    </Box>
  );
}
---
Platforms: All

Example 3: WRONG -- mixed text children (linter error: no-mixed-text-children)
---
// Creates multiple __TEXT__ nodes that overlap in layout
<Text style={{ fontSize: 14, color: '#cdd6f4' }}>
  Score: {score}
</Text>
---
Platforms: All

Example 4: RIGHT -- template literal instead of mixed children
---
<Text style={{ fontSize: 14, color: '#cdd6f4' }}>
  {`Score: ${score}`}
</Text>
---
Platforms: All

Example 5: RIGHT -- flexGrow for space-filling elements
---
<Box style={{ width: '100%', height: '100%', backgroundColor: '#1e1e2e' }}>
  <Header />
  <Box style={{ flexGrow: 1 }}>
    <MainContent />
  </Box>
  <Footer />
</Box>
---
Platforms: All

Example 6: RIGHT -- ScrollView with explicit height
---
<Box style={{ width: '100%', height: '100%' }}>
  <ScrollView style={{ height: 300, width: '100%' }}>
    <Text style={{ fontSize: 14 }}>Scrollable content</Text>
  </ScrollView>
</Box>
---
Platforms: All

=== PLATFORM NOTES ===
These 4 rules apply to all platforms. The layout engine shares fundamental constraints
across all targets (Love2D, web, terminal, etc.). Root sizing, space-filling, scroll
containers, and text composition are universal concerns independent of rendering backend.

=== CRITICAL RULES ===
1. Root containers MUST have width: '100%', height: '100%'
   Why: The root element IS the outermost container. Without explicit dimensions, it has zero
   size and nothing will be visible. flexGrow on the root has no parent to grow into.
   Linter: no-flexgrow-root (error)

2. Use flexGrow: 1 for space-filling elements
   Why: When you want an element to absorb remaining space after its siblings measure, use
   flexGrow: 1. This is the correct way to fill available space in a flex container. Never
   hardcode pixel heights to fill a known window size — that creates dead air at different
   resolutions.
   Linter: Enforced at build time

3. ScrollView needs explicit height
   Why: Scroll containers are excluded from auto-sizing. They need height: <number> or
   flexGrow: 1 to define their scroll viewport. Without explicit height, a ScrollView has
   zero size and no content is visible.
   Linter: Enforced at build time

4. No mixed text and expressions in Text -- use template literals
   Why: <Text>Score: {score}</Text> creates two separate __TEXT__ nodes ("Score: " and the
   value of score). The layout engine lays these out as separate flex items at y=0, causing
   them to overlap. Using {`Score: ${score}`} produces a single __TEXT__ node. This is a
   pending reconciler fix and remains the most reliable pattern.
   Linter: no-mixed-text-children (error)
