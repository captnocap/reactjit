=== OVERVIEW ===
Lua-owned interaction is an architectural pattern where complex interactive primitives (TextInput, TextEditor, VideoPlayer, ContextMenu, CodeBlock) are fully owned by Lua. ALL state — text content, cursor position, selection range, scroll offsets, playback state, control visibility, hover targets, drag state — lives entirely in Lua and is stored on the node itself (`node.inputState`, `node.editorState`, `node._vp`, etc.). The JS/TypeScript side only hears about boundary events: focus, blur, submit, changeText, timeUpdate, play, pause, ended. There is zero bridge traffic per keystroke, per cursor movement, per frame of video playback, per control interaction.

This pattern contrasts with traditional React controlled components where every keystroke triggers a state update, re-render, and reconciliation pass. In Lua-owned mode, React declares the component once (`<TextInput value="..." />`), Lua renders and manages all interaction, and JS only receives callbacks when the user is done editing (blur/submit) or when critical state changes (play/pause/ended). This eliminates thousands of bridge calls per second for high-interaction widgets.

The pattern is used for: TextInput (single/multiline text editing with cursor, selection, clipboard), TextEditor (multi-line code editor with syntax highlighting, line numbers, scroll), VideoPlayer (video playback with play/pause, seek, volume, fullscreen controls), ContextMenu (right-click menus with nested submenus, keyboard navigation), and CodeBlock (syntax-highlighted read-only code with line numbers, copy button).

=== API / SYNTAX ===
**How It Works**

1. **React declares the component**: `<TextInput value="initial" onChangeText={callback} />`
2. **Reconciler creates a host node**: `createElement('TextInput')` in `hostConfig.ts`
3. **Layout engine computes bounds**: `layout.lua` calculates `node.computed.x/y/w/h`
4. **Painter delegates to Lua module**: `painter.lua` calls `TextInput.draw(node)` in `textinput.lua`
5. **Lua module renders everything**: Text content, cursor, selection, placeholder, scrolling viewport, border, all drawn using `love.graphics.*` primitives
6. **Events go directly to Lua**: `events.lua` calls `TextInput.onKeyPressed(node, key)`, `TextInput.onTextInput(node, text)`, `TextInput.onPointerPressed(node, x, y)`, etc.
7. **Boundary events pushed to JS**: When editing is done (blur/submit), Lua calls `pushEvent({ type: 'changeText', targetId: node.id, payload: { text: is.text } })`, which triggers the React `onChangeText` callback.

**State Storage Convention**

Each Lua-owned primitive stores its state on the node:

- **TextInput**: `node.inputState` — `{ text, cursorPos, scrollX, selectStart, selectEnd, blinkTimer, blinkOn, isDragging, lastValue }`
- **TextEditor**: `node.editorState` — `{ lines, cursorLine, cursorCol, scrollY, scrollX, selectStartLine, selectStartCol, selectEndLine, selectEndCol, blinkTimer, blinkOn, isDragging, lastValue }`
- **VideoPlayer**: `node._vp` — `{ showControls, controlsTimer, hoverTarget, draggingSeek, draggingVolume, volume, muted, isFullscreen }`
- **ContextMenu**: `node._cm` — `{ hoverIndex, selectedPath, keyboardIndex }`
- **CodeBlock**: `node._cb` — `{ scrollY, hoverCopy }`

State is initialized on first draw (via `ensureState(node)` or similar) and persists across re-renders. When props change (e.g. `value` prop updated externally), the Lua module detects the change (`props.value ~= is.lastValue`) and syncs state.

**Controlled Value Pattern**

Lua-owned primitives support controlled values:

```tsx
const [text, setText] = useState('');

<TextInput
  value={text}
  onChangeText={setText}
/>
```

The Lua module checks on each draw: if `props.value ~= is.lastValue`, it updates `is.text = props.value` and `is.lastValue = props.value`. This allows external updates (e.g. "reset to empty") while preserving Lua-owned editing state between external changes.

**Event Flow**

Lua-owned primitives subscribe to events in `events.lua`:

```lua
if node.type == "TextInput" then
  TextInput.onKeyPressed(node, key, isRepeat)
  handled = true
elseif node.type == "TextEditor" then
  TextEditor.onKeyPressed(node, key, isRepeat)
  handled = true
-- ...
end
```

Events are handled directly in Lua. Boundary events are pushed to JS:

```lua
pushEvent({
  type = "changeText",
  payload = { type = "changeText", targetId = node.id, text = is.text },
})
```

The event dispatcher in `eventDispatcher.ts` routes these to React callbacks:

```ts
if (event.type === 'changeText') {
  const node = findNodeById(event.payload.targetId);
  if (node?.props?.onChangeText) {
    node.props.onChangeText(event.payload.text);
  }
}
```

**Rendering Integration**

The painter delegates to Lua modules:

```lua
if node.type == "TextInput" then
  TextInput.draw(node)
elseif node.type == "TextEditor" then
  TextEditor.draw(node)
elseif node.type == "VideoPlayer" then
  VideoPlayer.draw(node)
elseif node.type == "ContextMenu" then
  ContextMenu.draw(node)
elseif node.type == "CodeBlock" then
  CodeBlock.draw(node)
-- ...
end
```

Each module's `draw()` function renders the complete widget using `love.graphics.*` primitives: rectangles, text, lines, circles, stencils, scissors, etc.

**Lua Module Structure (Example: textinput.lua)**

```lua
local TextInput = {}

-- Init (inject dependencies: measure.lua for font resolution)
function TextInput.init(config)
  Measure = config.measure
end

-- State management
function TextInput.initState(node)
  node.inputState = { text = "", cursorPos = 0, ... }
end

local function ensureState(node)
  if not node.inputState then TextInput.initState(node) end
  return node.inputState
end

-- Rendering
function TextInput.draw(node)
  local is = ensureState(node)
  -- Sync controlled value
  local props = node.props or {}
  if props.value and props.value ~= is.lastValue then
    is.text = props.value
    is.lastValue = props.value
  end
  -- Render background, border, text, cursor, selection, placeholder...
end

-- Event handlers
function TextInput.onKeyPressed(node, key, isRepeat)
  local is = ensureState(node)
  -- Handle cursor movement, deletion, clipboard, submit...
end

function TextInput.onTextInput(node, text)
  local is = ensureState(node)
  -- Insert text at cursor...
end

function TextInput.onPointerPressed(node, x, y, button)
  local is = ensureState(node)
  -- Position cursor, start selection drag...
end

-- ... more handlers ...

return TextInput
```

**Creating a New Lua-Owned Primitive (Checklist)**

1. **Create Lua module** (`lua/<name>.lua`) with `init()`, `draw()`, event handlers.
2. **Add state field** to node (`node.<name>State`) initialized in `ensureState()` or similar.
3. **Register in painter** (`painter.lua`): add `elseif node.type == "MyWidget" then MyWidget.draw(node)`.
4. **Register in events** (`events.lua`): add event handlers for your widget type.
5. **Create TypeScript component** (`packages/shared/src/MyWidget.tsx`): declares host element, props interface, renders `React.createElement('MyWidget', props)`.
6. **Add host config entry** (`packages/native/src/hostConfig.ts`): add `'MyWidget'` to supported types, implement `createInstance()`, `commitUpdate()`, etc.
7. **Define props interface** (`packages/shared/src/types.ts`): add `MyWidgetProps` with all props and event callbacks.
8. **Export from index** (`packages/shared/src/index.ts`): `export { MyWidget } from './MyWidget';`.
9. **Sync to CLI runtime**: `make cli-setup` to copy Lua files to `cli/runtime/lua/`.
10. **Create storybook story** (`storybook/src/stories/MyWidgetStory.tsx`): demonstrate all features.

=== EXAMPLES ===
Example 1: TextInput - Zero bridge traffic per keystroke
---
```tsx
import { TextInput } from '@reactjit/core';
import { useState } from 'react';

function SearchBox() {
  const [query, setQuery] = useState('');

  return (
    <TextInput
      value={query}
      onChangeText={setQuery}
      placeholder="Search..."
      style={{ width: 300, height: 32 }}
    />
  );
}
```

**What happens:**
- User types "hello" (5 keystrokes).
- Each keystroke is handled in Lua (`textinput.lua:onTextInput`).
- Lua updates `node.inputState.text`, redraws cursor, no JS calls.
- User presses Enter or Tab (blur).
- Lua calls `pushEvent({ type: 'changeText', text: 'hello' })`.
- React receives `onChangeText('hello')` callback, `setQuery('hello')` updates state.
- **Total JS bridge calls: 1** (not 5+).

Compare to traditional React: Each keystroke would trigger `onChange`, state update, re-render, reconciliation, props diff, bridge call to update Lua — 5+ bridge calls per keystroke.
---
Platforms: Love2D

Example 2: VideoPlayer - Zero bridge traffic per frame
---
```tsx
import { VideoPlayer } from '@reactjit/core';
import { useState } from 'react';

function VideoDemo() {
  const [currentTime, setCurrentTime] = useState(0);

  return (
    <VideoPlayer
      src="demo.ogv"
      w={640}
      h={360}
      onTimeUpdate={(e) => setCurrentTime(e.currentTime)}
      onPlay={() => console.log('started')}
      onPause={() => console.log('paused')}
      onEnded={() => console.log('finished')}
    />
  );
}
```

**What happens:**
- User hovers over video, controls appear (Lua-managed, no JS call).
- User clicks play button (Lua handles click, calls `videos.lua:play()`, pushes `onPlay` event).
- Video plays at 30fps (Lua renders each frame, updates seek bar position, no JS calls).
- User drags seek bar (Lua handles drag, updates `node._vp.draggingSeek`, seeks video, no JS calls).
- Every 0.5s, Lua pushes `onTimeUpdate` event with current timestamp.
- User clicks pause (Lua pauses, pushes `onPause` event).
- **Total JS bridge calls during playback: ~2 per second** (timeUpdate events), not 30fps.

Compare to traditional React: Every frame would trigger state update for seek bar position, every control hover would trigger hover state update, every drag would trigger drag state update — hundreds of bridge calls per second.

Example 5: Building a custom Lua-owned slider
---
**Lua module (lua/customslider.lua):**
```lua
local CustomSlider = {}

function CustomSlider.initState(node)
  node.sliderState = {
    value = (node.props or {}).value or 0,
    dragging = false,
  }
end

local function ensureState(node)
  if not node.sliderState then CustomSlider.initState(node) end
  return node.sliderState
end

function CustomSlider.draw(node)
  local ss = ensureState(node)
  local c = node.computed
  local props = node.props or {}

  -- Sync controlled value
  if props.value and props.value ~= ss.value then
    ss.value = props.value
  end

  -- Draw track
  love.graphics.setColor(0.3, 0.3, 0.3, 1)
  love.graphics.rectangle('fill', c.x, c.y + c.h/2 - 2, c.w, 4)

  -- Draw thumb
  local thumbX = c.x + (ss.value / (props.max or 100)) * c.w
  love.graphics.setColor(0.2, 0.5, 1, 1)
  love.graphics.circle('fill', thumbX, c.y + c.h/2, 8)
end

function CustomSlider.onPointerPressed(node, x, y, button)
  local ss = ensureState(node)
  ss.dragging = true
  CustomSlider.updateValue(node, x)
end

function CustomSlider.onPointerMoved(node, x, y)
  local ss = ensureState(node)
  if ss.dragging then
    CustomSlider.updateValue(node, x)
  end
end

function CustomSlider.onPointerReleased(node, x, y, button)
  local ss = ensureState(node)
  ss.dragging = false
  -- Push final value to JS
  pushEvent({
    type = "sliderChange",
    payload = { type: "sliderChange", targetId = node.id, value = ss.value },
  })
end

function CustomSlider.updateValue(node, x)
  local ss = ensureState(node)
  local c = node.computed
  local props = node.props or {}
  local min = props.min or 0
  local max = props.max or 100
  local ratio = (x - c.x) / c.w
  ss.value = min + ratio * (max - min)
  ss.value = math.max(min, math.min(max, ss.value))
end

return CustomSlider
```

**TypeScript component (packages/shared/src/CustomSlider.tsx):**
```tsx
import React from 'react';

interface CustomSliderProps {
  value?: number;
  min?: number;
  max?: number;
  onChange?: (value: number) => void;
  style?: any;
}

export function CustomSlider(props: CustomSliderProps) {
  return React.createElement('CustomSlider', props);
}
```

**Register in painter (lua/painter.lua):**
```lua

  CustomSlider.draw(node)
```

**Register events (lua/events.lua):**
```lua

  CustomSlider.onPointerPressed(focusedNode, x, y, button)
  handled = true
```

**Usage:**
```tsx
import { CustomSlider } from '@reactjit/core';
import { useState } from 'react';

function VolumeControl() {
  const [volume, setVolume] = useState(50);

  return (
    <CustomSlider
      value={volume}
      min={0}
      max={100}
      onChange={setVolume}

    />
  );
}
```

User drags slider, Lua updates thumb position on every mouse move (no JS calls). On release, Lua pushes final value to JS via `onChange` callback. **Total bridge calls: 1** (not one per pixel dragged).

=== CRITICAL RULES ===
- **State lives in Lua, not React** — Never try to manage cursor position, scroll offsets, hover state, or drag state in React. It all belongs in Lua on `node.<name>State`.
- **Boundary events only** — Only push events to JS when the user is done (blur, submit, ended) or when critical state changes (play, pause, error). Do NOT push events per keystroke, per cursor move, per frame, per hover.
- **Controlled value sync on draw** — Check `props.value ~= state.lastValue` on every draw and sync if changed. This allows external updates (reset button, undo/redo) while preserving Lua-owned editing state between updates.
- **Event handlers return handled = true** — Event handlers in `events.lua` must return `true` to prevent event bubbling to parent nodes. Otherwise click-through bugs occur.
- **State persists across re-renders** — When React re-renders and calls `commitUpdate()`, Lua keeps the existing `node.<name>State` intact. Do NOT reinitialize state on every render.
- **Use ensureState() pattern** — Initialize state lazily on first draw/event, not in `createElement()`. This avoids state allocation for off-screen nodes.
- **Font/measure dependency injection** — Lua modules depend on `measure.lua` for font resolution. Inject via `<Module>.init({ measure = Measure })` in `init.lua`.
- **Painter delegation order matters** — Render Lua-owned primitives AFTER basic nodes but BEFORE overlays (focus rings, inspector). Check `painter.lua` for correct ordering.
- **Bridge event types must match** — Event type string in Lua (`pushEvent({ type: "changeText" })`) must match case in `eventDispatcher.ts` and prop name in `types.ts` (`onChangeText`).
- **Clipboard is Lua-owned** — TextInput/TextEditor handle Ctrl+C/X/V using `love.system.getClipboardText()` / `setClipboardText()`. No JS clipboard API involved.
- **Cursor blink timer is Lua-owned** — TextInput/TextEditor manage `blinkTimer` and `blinkOn` state in Lua, updated in main loop via `update(dt)` callback. No React state updates for animation.
- **ScrollView integration** — Lua-owned primitives can be children of ScrollView. The painter applies scissor/translate for scroll offset automatically.
- **Focus integration** — Lua-owned primitives participate in focus navigation. They must handle `onFocus`/`onBlur` events from `focus.lua` and render focus styles accordingly.
- **Performance is the goal** — The entire point of Lua-owned interaction is to eliminate bridge traffic. If you're pushing events per keystroke or per frame, you're doing it wrong.
