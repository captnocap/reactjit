=== OVERVIEW ===
Terminal is a non-visual capability node that manages a PTY (pseudo-terminal) session on the Lua side. It renders nothing — drop it in your React tree to own a shell session. Output arrives via events (onData, onDirtyRows, onCursorMove) and input is sent via `bridge.rpc('pty:write')`.

For most use cases, pair Terminal with the `usePTY` hook for managed state and convenient send helpers.

Terminal supports three session archetypes: `user` (standard user shell), `root` (elevated shell, prompts for sudo if needed), and `template` (fresh ephemeral PTY per command with clean environment).

This is the headless shell primitive. If you want a visual terminal with classified tokens and rendering, use SemanticTerminal instead.

=== API / SYNTAX ===
```tsx
import { Terminal } from '@reactjit/core';
```

Props:
| Prop         | Type                                                        | Default   | Description                                  |
|--------------|-------------------------------------------------------------|-----------|----------------------------------------------|
| type         | 'user' \| 'root' \| 'template'                             | 'user'    | Session archetype                            |
| shell        | string                                                      | 'bash'    | Shell executable                             |
| cwd          | string                                                      | undefined | Working directory for the child process      |
| env          | Record<string, string \| false>                             | undefined | Environment overrides (false to unset)       |
| rows         | number                                                      | 24        | Terminal rows                                |
| cols         | number                                                      | 80        | Terminal columns                             |
| session      | string                                                      | auto      | Named session ID for RPC targeting           |
| transport    | 'bridge' \| 'ws' \| 'http' \| 'tor'                        | 'bridge'  | Transport backend                            |
| autoConnect  | boolean                                                     | true      | Auto-spawn on mount                          |
| onData       | (event: { data: string }) => void                           | undefined | Raw PTY output chunks (ANSI-encoded)         |
| onDirtyRows  | (event: { rows: Array<{ row, text }> }) => void             | undefined | Structured row data from vterm               |
| onCursorMove | (event: { row, col, visible }) => void                      | undefined | Cursor position changes                     |
| onConnect    | (event: { shell, ptyType, session? }) => void               | undefined | Shell process started                        |
| onExit       | (event: { exitCode: number \| null }) => void               | undefined | Shell process exited                         |
| onError      | (event: { error: string }) => void                          | undefined | Spawn error                                  |

=== EXAMPLES ===
Example 1: Stream shell output
---
```tsx
const [output, setOutput] = useState('');

<Terminal
  type="user"
  onData={(e) => setOutput(prev => prev + e.data)}
/>
```
---
Platforms: love2d

Example 2: With usePTY hook (recommended)
---
```tsx
const { output, send, terminalProps } = usePTY({ type: 'user', session: 'main' });

<Terminal {...terminalProps} />
<Text style={{ fontSize: 12 }}>{output}</Text>
```
---
Platforms: love2d

Example 3: Template PTY with custom environment
---
```tsx
<Terminal
  type="template"
  env={{ MY_API: 'key123' }}
  session="cmd"
  onData={(e) => append(e.data)}
  onExit={(e) => setDone(true)}
/>
```
---
Platforms: love2d

=== PLATFORM NOTES ===
Love2D:
  Terminal spawns a real PTY process via Lua's pty.lua module. The vterm library parses ANSI sequences and exposes structured row data via onDirtyRows. Session IDs are used for RPC targeting — send input with `bridge.rpc('pty:write', { session, data })`.

=== CRITICAL RULES ===
- Terminal renders nothing. It is purely a capability node. Build your own visual representation using the event data, or use SemanticTerminal for a visual terminal.
- The `session` prop determines the RPC namespace. If omitted, an auto-generated ID is used. Always specify `session` if you need to send input via RPC.
- Setting env values to `false` unsets those environment variables in the child process.
