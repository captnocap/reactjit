=== OVERVIEW ===
ReactJIT embeds a QuickJS JavaScript runtime inside the Love2D process using LuaJIT FFI. The
bridge (lua/bridge_quickjs.lua on the Lua side, packages/renderer/src/NativeBridge.ts on the
JS side) is the transport layer between React and Lua. It handles three responsibilities:

1. Mutation commands (JS to Lua). When React commits changes, the reconciler serializes
   mutation commands as a JSON string and calls __hostFlush(jsonStr). Lua parses the JSON and
   applies the commands to the instance tree. JSON serialization is required to avoid a QuickJS
   GC race condition where large string properties get silently dropped during FFI object
   property traversal.

2. Event polling (Lua to JS). Lua queues input events (mouse, keyboard, touch, scroll, etc.)
   during love.update(). Each frame, Lua calls the JS function _pollAndDispatchEvents(), which
   reads the event queue via __hostGetEvents(). Events are dispatched to React handlers through
   the event dispatcher.

3. RPC (bidirectional). JavaScript can call Lua functions via bridge.rpc(method, args). This
   sends a command to Lua, which processes it and queues a response event. The JS side returns
   a Promise that resolves when the response arrives (typically next frame). Used by hooks like
   useLoveRPC.

The bridge creates a QuickJS runtime with a 1MB stack and 64MB heap. It registers three
host functions via C trampolines (because LuaJIT cannot create FFI callbacks that return
JSValue structs):

  __hostFlush(jsonStr) -- JS sends mutation commands to Lua
  __hostGetEvents()    -- JS polls for input events from Lua
  __hostMeasureText()  -- JS queries text dimensions from Love2D's font rasterizer

Value conversion between JS and Lua uses direct FFI traversal for small objects (events,
RPC responses) and JSON serialization for large payloads (mutation command batches). The
bridge auto-detects JSValue tag layout at initialization to handle different QuickJS builds.

The NativeBridge class on the JS side implements the IBridge interface: send(), flush(),
subscribe(), rpc(), setState(). It is immediately ready at startup (no async initialization)
because QuickJS globals are available from the moment the runtime is created.

=== API / SYNTAX ===
Host functions exposed to JavaScript (registered via C trampolines):

  __hostFlush(commands: string)
    Receives JSON-serialized mutation commands from the reconciler. Lua parses the JSON and
    forwards the command array to tree.applyCommands(). Called from resetAfterCommit() at the
    end of every React commit.

  __hostGetEvents(): any[]
    Returns the current event queue as a JavaScript array. Called once per frame from
    _pollAndDispatchEvents(). The queue is cleared after reading.

  __hostMeasureText(params: object): object
    Synchronous text measurement. JS passes font size, text content, and wrap constraint;
    Lua measures using Love2D's font rasterizer and returns { width, height, lines }.

  __hostLog(message: string)
    Routes console.log/warn/error from JS to Lua's terminal output.

  __hostReportError(error: object)
    Reports JS errors with name, message, and stack trace to Lua's error overlay.

  __hostRandomBytes(count: number): number[]
    Reads random bytes from /dev/urandom for the crypto.getRandomValues() polyfill.

NativeBridge methods (JS side):

  bridge.send(type, payload?)  -- Queue a command for Lua
  bridge.flush()               -- Send all queued commands immediately
  bridge.subscribe(type, fn)   -- Listen for events from Lua (returns unsubscribe function)
  bridge.rpc(method, args?)    -- Call a Lua function, returns Promise
  bridge.setState(key, value)  -- Push a state update to Lua

Event flow:
  Lua queues events -> JS polls via __hostGetEvents -> bridge dispatches to subscribers
  -> eventDispatcher routes to React handlers via handlerRegistry

RPC flow:
  JS sends 'rpc:call' with { id, method, args } -> Lua processes the call
  -> Lua queues 'rpc:{id}' response -> JS resolves the Promise

=== EXAMPLES ===
Example 1: Using bridge.rpc from a component via useLoveRPC
---
import { useState, useEffect } from 'react';
import { Box, Text } from '@reactjit/core';
import { useLoveRPC } from '@reactjit/core';
import { useThemeColors } from '@reactjit/theme';

function SystemInfo() {
  const c = useThemeColors();
  const rpc = useLoveRPC();
  const [info, setInfo] = useState<string>('Loading...');

  useEffect(() => {
    rpc('system:info').then((result: any) => {
      setInfo(result.os + ' - ' + result.gpu);
    });
  }, []);

  return (
    <Box style={{ padding: 16 }}>
      <Text style={{ fontSize: 14, color: c.text }}>{`${info}`}</Text>
    </Box>
  );
}
---
Platforms: love2d

Example 2: Subscribing to bridge events
---
import { useEffect } from 'react';
import { useBridge } from '@reactjit/core';

function EventListener() {
  const bridge = useBridge();

  useEffect(() => {
    // Subscribe to a custom event from Lua
    const unsub = bridge.subscribe('custom:data', (payload: any) => {
      console.log('Received:', payload.value);
    });
    return unsub;
  }, [bridge]);

  return null;
}
---
Platforms: love2d

=== CRITICAL RULES ===
- Mutation commands are JSON-serialized before crossing the bridge. This is not optional -- it avoids a QuickJS GC race where large string properties get silently dropped during FFI object traversal.
- Events are polled, not pushed. Lua queues events; JS reads them once per frame. There is no callback mechanism from Lua to JS -- everything goes through the event queue.
- RPC responses arrive on the next frame (or later if Lua processing takes multiple frames). The returned Promise resolves when the response event arrives. Default timeout is 5 seconds.
- The QuickJS runtime has a 64MB heap limit. Exceeding this crashes the JS runtime. Keep React state lean -- large datasets belong in Lua (SQLite, localstore).
- Host functions use C trampolines because LuaJIT cannot create FFI callbacks that return JSValue structs. The C shim (qjs_ffi_shim.c) provides void-returning wrappers.
