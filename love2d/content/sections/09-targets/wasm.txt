=== OVERVIEW ===
The WASM target compiles Love2D to WebAssembly via love.js (an Emscripten port of Love2D). The result is the same Love2D rendering pipeline -- Lua layout engine, OpenGL painter, font rasterizer -- running inside a browser `<canvas>` element. This is not react-dom. There is no DOM rendering, no CSS. It is Love2D compiled to WASM.

The key difference from the native target is the transport layer:

- Native: JavaScript runs inside embedded QuickJS. Bridge uses C FFI (synchronous, in-process).
- WASM: Lua runs inside Emscripten-compiled Love2D. JavaScript runs natively in the browser. Bridge uses `Module.FS` file polling -- React writes mutation commands to a virtual filesystem file, Lua reads them each frame.

The React app uses `createWasmApp()` from `@reactjit/renderer` instead of `createLove2DApp()`. This factory creates a `WasmBridge` adapter that wraps `window.ReactJITBridge` (injected by `bridge.js`) as an `IBridge` for React.

The entry point file is `src/main-wasm.tsx` (or `src/main-web.tsx`). The CLI builds it with esbuild as IIFE format with `globalName: 'ReactJIT'`, same as the native target.

=== API / SYNTAX ===

---- Entry point ----------------------------------------------------------

```tsx
// src/main-wasm.tsx
import { createWasmApp } from '@reactjit/renderer';
import { App } from './App';

const app = createWasmApp();
app.render(<App />);
```

---- WasmBridge transport -------------------------------------------------

The bridge between browser JS and WASM Lua uses two communication channels:

1. Mutation commands (React -> Lua): React serializes commands as JSON and writes them to `__reconciler_in.json` via `Module.FS.writeFile()`. Lua reads this file each frame in its update loop.

2. Events (Lua -> React): The `window.ReactJITBridge` object provides subscribe/send/rpc methods. Events are queued by Lua and polled by JS through the bridge adapter.

The `WasmBridge` class buffers mutation commands if `Module.FS` is not yet ready (the Emscripten filesystem takes a few frames to initialize) and flushes them once available.

---- Build command --------------------------------------------------------

```bash
rjit build web             # Builds WASM target
```

This outputs `web/bundle.js` using the entry point priority: `main-wasm.tsx` > `main-web.tsx`.

---- History adapter for WASM ---------------------------------------------

WASM builds can use `createBrowserHistory()` from `@reactjit/router` to sync the router with the browser's URL bar:

```tsx
import { createBrowserHistory } from '@reactjit/router';

const history = createBrowserHistory();
// history.push('/page') updates window.location via pushState
// browser back/forward buttons trigger popstate events
```

For WASM apps that don't need URL bar integration, `createMemoryHistory()` works identically to native.

import { Box, Text } from '@reactjit/core';

  const colors = useThemeColors();
  const { width, height } = useWindowDimensions();

    <Box style={{ width: '100%', height: '100%', backgroundColor: colors.bg, justifyContent: 'center', alignItems: 'center' }}>
      <Text style={{ color: colors.text, fontSize: 18 }}>{`${width} x ${height}`}</Text>
    </Box>
  );
}

=== PLATFORM NOTES ===
Love2D:
  The WASM target uses love.js, which compiles Love2D with Emscripten. The rendering is still OpenGL (via WebGL translation). Performance is generally lower than native due to the WASM overhead and WebGL translation layer.

  `Module.FS` is Emscripten's virtual filesystem. Files written via `Module.FS.writeFile()` exist only in memory -- they are not persisted to disk. The bridge writes and reads from the Love2D save directory within this virtual filesystem.

  The `WasmBridge` polls for `Module.FS` availability on startup. The Emscripten filesystem may not be ready for the first few frames after page load. Commands are buffered until the save directory is resolved.

  Multi-window is not supported in WASM builds. The `<Window>` component requires subprocess spawning, which is not available in the browser.

  love.js uses PUC Lua 5.1 (not LuaJIT). The layout engine and all Lua modules are compatible with both runtimes. The main impact is performance -- LuaJIT is significantly faster for the layout pass.

=== CRITICAL RULES ===
- The WASM target is still Love2D rendering to a canvas -- it is NOT react-dom, NOT CSS, NOT HTML. The same primitives (Box, Text, Image) and the same layout engine apply.
- Use `createWasmApp()` for the WASM entry point, not `createLove2DApp()`. They use different bridge adapters.
- `createBrowserHistory()` only works in WASM builds where `window.history` exists. Using it in a native Love2D app will crash because there is no `window` object.
- Multi-window (`<Window>` component) is not available in WASM builds.
- The entry point priority for the web target is `main-wasm.tsx` then `main-web.tsx`. The CLI checks both in order.
- App components (everything under `src/` except the entry point) should be identical between native and WASM. The entry point is the only file that differs.
