## Group 7: Entry (a039-a042) vs emit/entrypoints.js

### a039_app_init — MISMATCH
Four differences from monolith `emitRuntimeEntrypoints()` _appInit:
1. **Missing `luajit_runtime.registerHostFn` for OA unpack.** Monolith registers each OA setter with both `qjs_runtime` AND `luajit_runtime`. Atom only registers with `qjs_runtime`.
2. **Missing `mapBackend === 'lua_runtime'` skip.** Monolith skips Lua-backend maps in the `_rebuildMapN()` init loop. Atom would call rebuild on Lua-backend maps (which should only rebuild via LuaJIT).
3. **Missing `qjs_runtime.evalLuaMapData(...)` calls.** Monolith evaluates source data expressions and passes results to LuaJIT. Atom omits these.
4. **Missing `luajit_runtime.callGlobal("__rebuildLuaMaps")` at init.** Monolith calls the Lua master rebuild after data eval. Atom's wrapper registration block doesn't trigger the initial rebuild.

### a040_app_tick — MISMATCH
Two differences from monolith _appTick:
1. **Missing `_luaDataEvalBlock`.** Same issue as a038 — evalLuaMapData calls omitted before __rebuildLuaMaps.
2. **Collapsed code paths.** Monolith has three branches (hasDynStyles / maps||luaMaps / state-only). Atom collapses first two into `if (meta.hasDynStyles || ctx.maps.length > 0 || hasLuaMaps)`. Functionally similar but structurally different output.

### a041_app_exports — MATCH
All `app_get_*` exports and state ABI exports identical.

### a042_app_main — MATCH

### Pattern: evalLuaMapData is systematically missing
a003, a038, a039, and a040 all lack `qjs_runtime.evalLuaMapData()` calls. This suggests the Lua map data bridge was added to the monolith after the atoms were extracted. Any app with Lua-runtime maps would fail to pass source data from JS to Lua if atoms were used instead of monoliths.
