# Dogfood Roadmap: Smith/Forge Generating Framework Modules

## What tsz-gen Did and How

The `archive/tsz-gen/` directory contains a complete system where the **entire runtime** was authored in `.tsz` and compiled to `.zig` via the old Zig-based compiler. The evidence:

**Source files** lived in `runtime/tsz/` as `.mod.tsz`:
- `layout.mod.tsz` → `runtime/compiled/layout.zig`
- `state.mod.tsz` → `runtime/compiled/state.zig`
- `events.mod.tsz` → `runtime/compiled/events.zig`
- Plus ~70 more: `gpu.zig`, `compositor.zig`, `canvas.zig`, `sqlite.zig`, `crypto.zig`, `http_client.zig`, `wireguard.zig`, etc.

**The pipeline** was `tsz compile-runtime`:
1. Read `.mod.tsz` source (imperative TypeScript — no JSX, no App function)
2. Lex into tokens (same lexer as app compilation)
3. Detect imperative mode (no JSX → route to `modulegen.zig` instead of `codegen.zig`)
4. `modulegen.zig` orchestrated four sub-generators:
   - `typegen.zig` — enum, struct, tagged union, type alias declarations
   - `stmtgen.zig` — function bodies (control flow, variable declarations)
   - `exprgen.zig` — expressions within statements
   - `emitImports()` — `import { X } from './events'` → `const events = @import("events.zig")`
5. Output a complete `.zig` file with header, imports, types, module-level vars, and functions

**What `.mod.tsz` syntax looked like** (from the generated output and CLAUDE.md):
```tsx
// layout.mod.tsz — imperative, no JSX
import { EventHandler } from './events';

enum FlexDirection { row, column }
enum JustifyContent { start, center, end, space_between, space_around, space_evenly }

interface Style {
  width: ?f32 = null;
  height: ?f32 = null;
  flex_direction: FlexDirection = .column;
  // ...
}

interface Node {
  style: Style = {};
  children: []Node = &.{};
  computed: LayoutRect = {};
  text: ?[]const u8 = null;
  // ...
}

export function performLayout(node: *Node, parentW: f32, parentH: f32): void {
  // Pure algorithmic code — if/else, while, for, math
}
```

The old compiler (`modulegen.zig` + `typegen.zig` + `stmtgen.zig` + `exprgen.zig`) was ~25k lines of Zig and handled: enums, structs (via `interface`), tagged unions, function pointers, optionals, fixed arrays, slices, if/else, while, for, switch, break/continue/return, defer, try/catch, ternary, null coalescing, `@builtin()` passthrough, address-of, deref, ranges, camelCase→snake_case.

**Key design principle** from tsz-gen CLAUDE.md: "If it's not generating code, it should be generated code. The compiler is the ONLY hand-written Zig."

## What Smith Can Do Today

Smith (3,757 lines of JS across 5 files) is a **JSX-to-Zig compiler for app cartridges**. It takes `.app.tsz` files containing React-like components and produces `generated_*.zig` files that link against the framework.

### Smith's current capabilities:
- **JSX parsing**: `<Box>`, `<Text>`, `<Pressable>`, `<ScrollView>`, `<Image>`, `<Cartridge>`, plus HTML tag mapping (`<div>` → Box, `<span>` → Text, etc.)
- **Component system**: `function Name({props}) { return (<JSX>) }` — collected, inlined at call sites
- **State**: `useState(initial)` → compile-time state slots with `state.getSlot(N)` / `state.setSlot(N, val)` — supports int, float, boolean, string
- **Object arrays**: `useState([{field: val}])` → flat column-oriented arrays with QuickJS unpack bridge
- **Event handlers**: `onPress={() => setter(expr)}` → Zig function pointers + QuickJS eval for complex logic
- **Dynamic text**: Template literals → `std.fmt.bufPrint` with runtime state interpolation
- **Dynamic styles**: Ternary expressions in style blocks → runtime `dynStyles` updates
- **Dynamic colors**: Same ternary support for color properties
- **Conditionals**: `{cond && <Node>}`, `{cond ? <A> : <B>}` → display toggle via `_cond_N` arrays
- **`.map()` iteration**: Array mapping with per-item node pools, dynamic text per item, map-scoped handlers
- **Classifiers**: `.cls.tsz` imports with `C.Name` usage → merged style/node fields
- **`<script>` blocks**: Raw JS passed through to QuickJS runtime, function names tracked for handler dispatch
- **Lua handler output**: Parallel `luaParseHandler` / `luaParseValueExpr` paths for LuaJIT logic runtime
- **Import resolution**: `from "./file"` → reads `.tsz` and `.cls.tsz` files, routes content to `__scriptContent` / `__clsContent`

### Smith's output structure:
```zig
//! Generated by tsz compiler
const layout = @import("framework/layout.zig");
const Node = layout.Node;
const Style = layout.Style;
const Color = layout.Color;
const state = @import("framework/state.zig");
// ...

// State manifest
// slot 0: count (int)

// Generated node tree
var _arr_0 = [_]Node{ ... };
var _root = Node{ .children = &_arr_0 };

// Dynamic text buffers
var _dyn_buf_0: [64]u8 = undefined;

// Event handlers
fn _handler_press_0() void { ... }

// Map pools (if .map() used)
// Object array infrastructure (if useState([{}]) used)

// Entry points
pub fn app_get_root() *Node { return &_root; }
pub fn app_get_init() ... { ... }
pub fn app_get_tick() ... { ... }
```

### What Smith does NOT handle:
- **No imperative mode** — it requires `function App() { return (<JSX>) }`. No App function → `"Smith error: no App function found"`
- **No type declarations** — no `enum`, `interface`/`struct`, `union`, `type alias`
- **No pure functions** — can't emit standalone `pub fn foo()` outside the app scaffold
- **No module-level variables** — only state slots and dynamic buffers
- **No `@import` generation** — it hardcodes `@import("framework/layout.zig")` etc.
- **No `export`** — output is a self-contained app, not a library module
- **No `declare`** — no FFI declarations
- **No complex expressions** — expression parsing (`parseValueExpr`) handles basic arithmetic, comparisons, and ternaries but not arbitrary statement-level code
- **No control flow in function bodies** — no while, for, switch, defer, try/catch

## The Gap

| Capability | tsz-gen modulegen | Smith today | Gap |
|---|---|---|---|
| Imperative mode (no JSX) | ✅ Full | ❌ Requires App + JSX | **Critical** |
| Type declarations (enum) | ✅ `typegen.zig` | ❌ None | **Critical** |
| Struct declarations (interface) | ✅ `typegen.zig` | ❌ None | **Critical** |
| Tagged unions | ✅ `typegen.zig` | ❌ None | Medium |
| Function bodies (control flow) | ✅ `stmtgen.zig` | ❌ Only handler expressions | **Critical** |
| Expression codegen | ✅ `exprgen.zig` (full) | Partial (arithmetic, ternary) | High |
| Import → @import | ✅ `emitImports` | ❌ Hardcoded framework paths | High |
| Module-level vars | ✅ `emitModuleVars` | ❌ None | High |
| pub/export | ✅ Yes | ❌ No | High |
| JSX → Node tree | ✅ `codegen.zig` | ✅ Smith (better, JS-based) | — |
| State system | ✅ Yes | ✅ Yes (richer: object arrays) | — |
| camelCase → snake_case | ✅ Automatic | Partial (in rules.js tables) | Low |
| @builtin passthrough | ✅ Yes | ❌ No | Medium |
| Optionals (?T), pointers | ✅ Yes | ❌ No | Medium |
| Error handling (try/catch) | ✅ Yes | ❌ No | Medium |
| Zig string manipulation | ✅ Yes (std.mem, etc.) | ❌ No | Medium |

**The fundamental gap**: Smith is a JSX-to-node-tree compiler. Framework modules are imperative Zig libraries with no JSX. Smith needs a second mode — call it "module mode" or "imperative mode" — that produces `.zig` library files instead of app scaffolds.

## The Key Architectural Insight

Smith doesn't need to replicate everything the old modulegen did. The old compiler was a monolith — it compiled **everything** from `.tsz` to `.zig`, including the algorithmic guts of layout, state machines, crypto, etc.

Smith's strength is the **split-brain architecture**: Zig kernel (Forge) + JS intelligence (Smith) + QuickJS runtime + LuaJIT logic. For framework modules, this means:

- **Type definitions** (enums, structs) → Smith generates Zig type declarations
- **Pure algorithmic functions** (layout, hit testing, math) → Stay as Lua in QuickJS/LuaJIT, OR Smith generates Zig from imperative `.tsz`
- **Bridge code** (function signatures, FFI glue) → Smith generates the Zig interface
- **JSX-containing modules** (devtools, inspector) → Already handled by Smith's existing path

The pragmatic approach: framework modules that are mostly **type definitions + bridge glue** are easy wins. Modules that are mostly **algorithms** (layout, text measurement, canvas math) are harder and may not need Smith at all — they could stay hand-written Zig or be authored in Lua where the logic actually runs at runtime.

## Phased Plan

### Phase 0: Module Mode Detection in Forge

Add a `--module` flag or auto-detect `.mod.tsz` extension in `forge.zig`. When module mode is detected, Smith's `compile()` function takes a different path — no App function search, no JSX root parsing.

**Files to change**: `forge.zig` (flag handling), `smith/index.js` (new entry point `compileModule()`)

### Phase 1: Type Declarations

Add `emitTypes()` to Smith that walks tokens and generates:
- `enum Name { a, b, c }` → `pub const Name = enum { a, b, c };`
- `interface Name { field: Type = default; }` → `pub const Name = struct { field: Type = default };`
- `type Alias = ...` → `pub const Alias = ...;`

This is pure token walking — no scope analysis, no expression evaluation. The old `typegen.zig` is the reference for what patterns to handle.

**New file**: `smith/types.js` (~200-300 lines)

### Phase 2: Import Resolution

Extend Smith's import handling to generate proper `@import` statements:
- `import { Foo, Bar } from './events'` → `const events = @import("events.zig"); const Foo = events.Foo; const Bar = events.Bar;`
- `import baz from './baz'` → `const baz = @import("baz.zig");`

Smith already does simple import scanning in `forge.zig` for `.cls` and script content. This extends that to produce Zig import declarations.

**Files to change**: `smith/emit.js` (new `emitModuleImports()`), possibly `smith/index.js`

### Phase 3: Module-Level Variables

Emit `var` and `const` declarations at module scope:
- `const MAX_SLOTS: usize = 256;` → passthrough with type annotation
- `var slots: [MAX_SLOTS]StateSlot = undefined;` → passthrough

This is mostly literal passthrough with camelCase→snake_case conversion.

**Files to change**: `smith/emit.js`

### Phase 4: Function Signatures and Simple Bodies

Generate `pub fn name(params) ReturnType { ... }` for exported functions. Start with:
- Function signature parsing (parameter names, types, return type)
- Simple expression bodies (return, assignments, arithmetic)
- `pub` from `export` keyword

This is the hard part. The old `stmtgen.zig` + `exprgen.zig` were ~thousands of lines. Start with the subset needed for the easiest modules.

**New file**: `smith/stmt.js` (~500+ lines, growing)

### Phase 5: Control Flow

Add to `smith/stmt.js`:
- `if / else` → `if (...) { ... } else { ... }`
- `while` → `while (...) { ... }`
- `for (const x of arr)` → `for (arr) |x| { ... }`
- `switch` → `switch (...) { ... }`
- `return`, `break`, `continue`

### Phase 6: Advanced Expressions

- `@builtin()` passthrough
- Optional unwrap (`??`, `orelse`)
- Error handling (`try`, `catch`)
- Pointer operations (`&`, `.*`)
- Slice operations (`[start..end]`)
- Struct initialization (`.{ ... }`)

### Phase 7: Full Module Output

Wire everything together into `compileModule()`:
1. Header
2. Imports
3. Type declarations
4. Module-level vars
5. Functions
6. Return complete `.zig` source

Test against the archived `tsz-gen/runtime/compiled/` files — the generated output should match (or improve upon) the old compiler's output.

## Which Modules to Start With (Not Layout)

Ordered by increasing difficulty:

### 1. `events.zig` (60 lines of hand-written Zig) — EASIEST
- 1 struct (`EventHandler` — 10 optional function pointer fields)
- 2 small functions (`hitTest`, `hasHandlers`)
- No imports except `layout.zig`
- No complex types, no allocations, no error handling
- **Why first**: It's tiny, validates type emission + function signatures + basic control flow (if/while/return)

### 2. `easing.zig` (~80 lines estimated)
- Pure math functions: `easeInQuad(t)`, `easeOutCubic(t)`, etc.
- No state, no imports, no types beyond `f32`
- **Why second**: Validates pure function codegen with arithmetic expressions

### 3. `state.zig` (300+ lines)
- Structs: `ArraySlot`, `StringArraySlot`, `StateSlot`, `Value` (tagged union)
- Module-level vars with complex initializers
- Functions with allocator usage, error handling
- **Why third**: Validates tagged unions, module vars, `try`/`catch`, slice operations

### 4. `breakpoint.zig`, `classifier.zig`, `context_menu.zig` (small utility modules)
- Mix of structs + functions
- Good for exercising the pipeline on diverse patterns

### 5. `layout.zig` (1602 lines) — LAST, HARDEST
- 15 enum types, 5 struct types (Style: ~60 fields, Node: ~100 fields)
- Complex algorithm functions with nested loops, float math, caching
- `layout.zig` is the boss fight — don't attempt until Phases 1-6 are solid

## What `.mod.tsz` Syntax Might Look Like

The old tsz-gen already established the syntax. Smith should follow it:

```tsx
// state.mod.tsz — imperative module, no JSX

// Imports use the same syntax as app .tsz
import { Node } from './layout';

// Enums
enum ValueKind { int, float, boolean, string }

// Structs (using 'interface' keyword as in the old compiler)
interface StateSlot {
  value: Value;
  dirty: bool = false;
}

// Tagged unions
union Value {
  int: i64;
  float: f64;
  boolean: bool;
  string: struct { buf: [256]u8, len: u8 };
}

// Module-level variables
const MAX_SLOTS: usize = 256;
var slots: [MAX_SLOTS]StateSlot = undefined;
var slot_count: usize = 0;

// Exported functions
export function createSlot(initial: i64): usize {
  const id = slot_count;
  std.debug.assert(id < MAX_SLOTS);
  slots[id] = .{ .value = .{ .int = initial }, .dirty = false };
  slot_count += 1;
  return id;
}

export function getSlot(id: usize): i64 {
  switch (slots[id].value) {
    .int: |v| return v;
    .float: |v| return @intFromFloat(v);
    .boolean: |v| return v ? @as(i64, 1) : 0;
    .string: return 0;
  }
}

// @builtin passthrough for Zig-specific operations
export function markDirty(): void {
  _dirty = true;
}
```

**Key syntax rules** (inherited from tsz-gen):
- `.mod.tsz` extension triggers imperative mode — no App function expected
- `interface` → Zig `struct` (not `class`, not `type`)
- `enum` → Zig `enum`
- `union` → Zig tagged `union(enum)`
- `export` → `pub`
- Type annotations are mandatory on function parameters and return types
- `@builtin()` calls pass through literally to Zig
- `const`/`var` at module level emit as-is
- Body syntax is TypeScript-flavored but maps 1:1 to Zig control flow

**What stays in Lua/QuickJS**: Runtime behavior that's already in `<script>` blocks or the QuickJS bridge doesn't need `.mod.tsz`. Layout **algorithms** could stay Lua and be called via the bridge. But type definitions and the Zig interface (the Node struct, the Style struct, the public API surface) should be `.mod.tsz` so there's one source of truth.

## The Split: Types in .mod.tsz, Algorithms Stay Where They Are

The realistic near-term target is NOT "rewrite layout.zig in .mod.tsz". It's:

1. **Type definitions** (Style, Node, enums) → `.mod.tsz` (Smith generates the Zig structs)
2. **Simple utility functions** (hitTest, hasHandlers, clampVal) → `.mod.tsz`
3. **Complex algorithms** (performLayout, estimateIntrinsicWidth, flex wrap logic) → Either stay as hand-written Zig OR move to Lua/QuickJS where they run at runtime anyway

The old tsz-gen tried to compile **everything** through one pipeline. That's why it needed 25k lines of compiler. Smith can be smarter — compile the **interface layer** and let the algorithmic guts live where they naturally belong (Zig for perf-critical, Lua for iteration-friendly).

## Validation Strategy

For each module converted to `.mod.tsz`:
1. Compile with `forge build --module module.mod.tsz`
2. Diff output against the current hand-written `.zig`
3. Build the full app (`zig build app`) to verify the generated module links correctly
4. Run the conformance suite to verify no behavioral regressions

The archived `tsz-gen/runtime/compiled/*.zig` files serve as additional reference — they show what the old compiler produced from the same logical source.
