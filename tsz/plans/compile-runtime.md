# `tsz compile-runtime` — Pre-compile .tsz to Runtime Zig Fragments

## RULE: Never Edit Compiled Zig — Edit the .tsz Source

The `.zig` output from `tsz compile-runtime` is **minified JS you would never edit in a browser.**
It is a build artifact. The `.tsz` source file is the source of truth. ALWAYS.

If the compiled output doesn't look right:
- **Fix the .tsz source** and recompile
- **Do NOT** hand-edit the generated .zig to tweak pixel values or adjust layout
- If the compiler isn't generating what you expect, that's a compiler bug — fix the compiler

The only exception: if there's a genuine layout engine bug (layout.zig), fix that separately
with proper justification (see the death note on layout.zig).

The .tsz source files MUST be kept alongside their compiled output. Deleting the .tsz
and keeping only the .zig is forbidden — you lose the ability to iterate.

```
tsz/devtools/InspectorOverlay.tsz    ← SOURCE OF TRUTH — edit this
tsz/runtime/compiled/framework/inspector_overlay.zig  ← BUILD ARTIFACT — never edit
```

## What This Is

A new CLI command that compiles a `.tsz` file into an embeddable Zig fragment — node definitions, style update functions, draw calls — suitable for freezing into the runtime. NOT a full app (no main loop, no SDL init, no event loop).

This is how framework overlays (inspector, devtools, crash screen) are authored in `.tsz` but deployed as frozen Zig modules in the runtime.

## The Pattern (Already Proven)

`inspector.zig` has a `renderOverlay()` function that was hand-translated from `.tsz` source:

```zig
// Source .tsz (comment in inspector.zig):
//   {hasHover() && <Box style={{ position:'absolute', backgroundColor:'#3b82f620',
//     borderWidth:1, borderColor:'#3b82f6a0' }} />}

// Compiled to:
pub fn renderOverlay() void {
    if (hover_valid) {
        gpu.drawRect(hover_x, hover_y, hover_w, hover_h,
            0.231, 0.510, 0.965, 0.125,
            0, 1,
            0.231, 0.510, 0.965, 0.627);
    }
}
```

This command **automates that hand-translation**.

## Usage

```bash
# Compile a .tsz overlay for the framework runtime
tsz compile-runtime inspector-overlay.tsz --framework
# Output: tsz/runtime/compiled/framework/inspector_overlay.zig

# Compile a user's custom runtime module
tsz compile-runtime my-overlay.tsz --user
# Output: tsz/runtime/compiled/user/my_overlay.zig

# Default (no flag) → user
tsz compile-runtime my-thing.tsz
# Output: tsz/runtime/compiled/user/my_thing.zig
```

## What It Generates (vs `tsz build`)

| | `tsz build` (full app) | `tsz compile-runtime` (fragment) |
|--|----------------------|-------------------------------|
| main() | Yes | No |
| SDL_Init / window creation | Yes | No |
| Event loop | Yes | No |
| State slots | Yes — global | Yes — namespaced to avoid conflicts |
| Node tree | Yes — `var root = Node{...}` | Yes — `pub fn buildTree() []Node` |
| Dynamic text buffers | Yes | Yes |
| Handlers | Yes — `fn _handler_press_N()` | Yes — same |
| useEffect | Yes — wired into loop | Yes — `pub fn tickEffects()` callable |
| Imports (layout, state, etc.) | Yes | Yes |
| Entry point | `pub fn main()` | **`pub fn render()` + `pub fn tick()` + `pub fn init()`** |

## Implementation

### Step 1: Add `compile-runtime` subcommand

**File: `tsz/compiler/main.zig`**

Add to the CLI action dispatch (alongside `build`, `run`, `dev`, `test`):

```zig
} else if (std.mem.eql(u8, action, "compile-runtime")) {
    return compileRuntime(args);
}
```

### Step 2: Modify codegen for fragment mode

**File: `tsz/compiler/codegen.zig`**

Add a mode flag to `Generator`:
```zig
mode: enum { full_app, runtime_fragment } = .full_app,
```

When `mode == .runtime_fragment`, `emitZigSource()` changes:
- **Skip:** main(), SDL init, window creation, event loop, loop template
- **Emit:** `pub fn init(base: usize)`, `pub fn tick()`, `pub fn render()`
- **State slots:** offset by `slot_base` parameter so they don't conflict with the host app's slots
- **Imports:** use relative paths (`@import("../layout.zig")` instead of `@import("layout.zig")`)

### Step 3: Output directory

```
tsz/runtime/compiled/
  framework/     — ships with tsz (inspector, crash screen, devtools)
  user/          — user's pre-compiled modules
```

The `--framework` flag writes to `framework/`, `--user` (default) writes to `user/`.

Filename: kebab-to-snake conversion. `inspector-overlay.tsz` → `inspector_overlay.zig`.

### Step 4: State Slot Namespacing

The critical difference from a full app: the runtime fragment's state slots must not collide with the host app's slots.

**Approach:** The fragment receives a `slot_base` at init time. All `state.createSlot()` calls in the fragment add `slot_base` to their IDs:

```zig
// In the fragment:
pub fn init(base: usize) void {
    slot_base = base;
    _ = state.createSlot(0);  // gets slot_base + 0
    _ = state.createSlot(0);  // gets slot_base + 1
}

// In the host (compositor or generated app):
const INSPECTOR_SLOTS = 3;  // number of slots the fragment needs
inspector.init(state.slotCount());  // start after app's slots
```

### Step 5: Integration Points

The host (compositor, generated app) calls:
```zig
// At init:
inspector.init(state.slotCount());

// Each frame:
inspector.tick(dt);

// After painting app:
inspector.render();
```

## Why This Matters

Without this, every runtime overlay is hand-translated from .tsz to Zig — error-prone and hard to maintain. With this, you author in .tsz (familiar syntax, uses the framework's own primitives), compile once, and the output is a frozen Zig module that the runtime calls directly.

Framework authors write `.tsz`. Users write `.tsz`. The runtime is `.zig`. The compiler bridges the gap automatically.
